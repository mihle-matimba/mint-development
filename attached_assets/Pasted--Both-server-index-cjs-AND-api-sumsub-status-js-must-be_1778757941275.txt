⚠️ Both server/index.cjs AND api/sumsub/status.js must be deployed. The Express version runs locally; the Vercel version runs in production. They are functionally equivalent and both received the same fix.

🧠 The problem we fixed (one-line summary)
During onboarding, address, ID number, tax, bank, mandate, etc. were saved correctly to user_onboarding. But profiles.phone_number, profiles.id_number, and profiles.address were not being populated from the data Sumsub captured during KYC. Phone in particular was never collected anywhere except inside Sumsub, so it never made it into profiles.

The root cause was that the Sumsub status endpoint never copied data from the Sumsub applicant record back into profiles. We added that copy ("writeback"), but had to handle three sub-issues to make it actually fire:

Authorization — server can't trust a random caller saying "update user X's profile", so we gated the writeback on a verified Bearer token where authedUserId === userId.
Frontend wasn't sending the token — the polling hooks were calling the status endpoint anonymously, so the gate always rejected them.
Already-verified users took an early exit path — if the KYC pack was already stored in user_onboarding_pack_details, the route returned {status: "verified"} immediately and skipped the writeback entirely. We had to add the writeback to that early-return path too.
📄 File 1: server/index.cjs (local Express server — ~lines 1261–1620)
What this file is
The local Express server that runs via npm run dev (node server/index.cjs). It mirrors the Vercel api/ handlers but in one big file. During local development, all /api/* calls hit this server.

What we changed
Change A — Optional Bearer auth at the top of /api/sumsub/status (~line 1272)

Before the route does anything, it checks for an Authorization: Bearer <jwt> header. If present, it validates the JWT against Supabase and stores the result in authedUserId. If absent or invalid, authedUserId stays null — the route still works for anonymous callers (so we don't break webhook flows or anything else), but the writeback is skipped.

let authedUserId = null;
if (req.headers.authorization?.startsWith("Bearer ")) {
  const token = req.headers.authorization.replace("Bearer ", "");
  const { data, error } = await supabase.auth.getUser(token);
  if (!error && data?.user) authedUserId = data.user.id;
}

Change B — Sumsub→profiles writeback in the GREEN handler (~line 1463)

When Sumsub returns a fresh GREEN verdict (i.e., the user just finished KYC), after we store the pack into user_onboarding_pack_details, we now also extract id_number, phone_number, and address from the applicant payload and update profiles — but only filling blanks (we never overwrite data the user typed in). Gated on authedUserId === userId.

Change C — Same writeback in the early-return short-circuit (~line 1294)

When user_onboarding_pack_details already has a row for this user, the route used to return immediately. We now load the stored pack from that row, run the same writeback on it, then return. This is what makes the fix work for users who already completed KYC before this fix shipped (like Lindelwa).

Change D — Tightened ID overwrite rule

id_number is only overwritten if the existing value isn't a valid 13-digit SA ID (so we never clobber a manually entered correct ID with garbage from Sumsub).

📄 File 3: api/_lib/supabase.js
What this file is
A small helper module that exports two ready-to-use Supabase clients (supabase = anon, supabaseAdmin = service role) and a authenticateUser(req) function that validates a Bearer JWT and returns { user, error }.

What we changed
Nothing in code. But api/sumsub/status.js now imports authenticateUser from it. Just confirm this file exists on your dev branch and exports authenticateUser — if it doesn't, copy it from this codebase. The function looks like this:

export async function authenticateUser(req) {
  const authHeader = req.headers.authorization;
  if (!authHeader || !authHeader.startsWith("Bearer ")) {
    return { user: null, error: "Missing or invalid Authorization header" };
  }
  const token = authHeader.replace("Bearer ", "");
  const { data, error } = await supabase.auth.getUser(token);
  if (error || !data?.user) return { user: null, error: error?.message || "Invalid token" };
  return { user: data.user, error: null };
}

What this file is
The 6-step onboarding wizard React component. Each step writes its data to either user_onboarding, profiles, or both via API calls.

What we changed

These are safe to leave in for now (they don't log PII, only field names + row counts). Remove later if you want quieter logs.

Change B — Mandate accept now POSTs save-mandate immediately (around line 1489). Previously, the mandate text was only persisted when the user reached the final completion step. If they bailed out, the mandate-accepted state was lost. Now it saves the moment the user taps "Accept".

Change C — Bearer token added to the Sumsub status poll (~line 732). The page polls /api/sumsub/status while the Sumsub widget is open. We now grab the current Supabase session token and attach it as Authorization: Bearer ...:

const { data: { session } } = await supabase.auth.getSession();
const token = session?.access_token;
const res = await fetch("/api/sumsub/status", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    ...(token && { Authorization: `Bearer ${token}` }),
  },
  body: JSON.stringify({ userId: user.id }),
});

This is the critical change — without it, the server's auth gate rejects the writeback for this caller.

🧩 How it all fits together (end-to-end flow)
Scenario: a user opens the app and the status poll runs

useSumsubStatus.js (or UserOnboardingPage.jsx) calls POST /api/sumsub/status with a Bearer token.
The handler (api/sumsub/status.js in prod, server/index.cjs in dev) validates the token → authedUserId = user.id.
The handler checks user_onboarding_pack_details:
Already verified? → load the stored pack, run syncSumsubProfileFields(...), return verified.
Not yet? → call Sumsub API for live applicant data; if GREEN, store the pack, run syncSumsubProfileFields(...), mark verified.
syncSumsubProfileFields reads the user's current profiles row and writes id_number / phone_number / address only into blank fields (never overwrites real data).
The user's Profile page now shows their phone number, ID, and address all populated.
Auth gate logic (security):

No token → writeback is skipped, but the rest of the route works normally (preserves existing webhook + anonymous flows).
Token but for a different user → writeback skipped (logged as "caller not authenticated as this user").
Token matches userId in the body → writeback runs.
